 * point of the pulse range). Phase 5 D-25 RM-threading pattern.
 */
@utility mark-pulse {
  animation: mark-pulse 1.2s var(--ease-brand) infinite alternate;
}

@keyframes mark-pulse {
  from { opacity: 0.4; }

  }

## Deferred Issues

**Build verification deferred to orchestrator post-parallel-completion**

`npm run build` failed during this plan's execution due to a parallel-execution race in `scripts/copy-renders.ts` + `scripts/optimize-images.mjs`. Specifically:

- `scripts/copy-renders.ts` line 53 `rmSync(RENDERS_DST, { recursive: true })` wipes `public/renders/` at every prebuild
- When two parallel-mode agents both invoke `npm run build`, agent A's `rmSync` deletes `public/renders/lakeview/_opt/` while agent B's `optimize-images.mjs` is mid-write ŌĆö sharp throws `ENOTEMPTY` or `unable to open for write`

This is **NOT caused by this plan's changes**. Evidence:
- Task 1 (lhci devDep) cannot affect the image pipeline ŌĆö it adds a CI-only binary
- Task 2 (CSS-only edits to src/index.css) doesn't touch any build script
- Task 3 (.gitignore additions) doesn't touch any build script
- The same race occurs on a fresh `npm run build` retry after my changes are committed

Per the parallel-execution mode notes in the executor protocol: "The orchestrator validates hooks once after all agents complete." The orchestrator's post-parallel verifier will run `npm run build` once in a quiet window and confirm:
1. `mark-pulse` class compiles into `dist/assets/*.css` (proves `@utility` block is syntactically valid)
2. Bundle size unchanged from pre-Wave-1 baseline
3. `npx tsx scripts/check-brand.ts` 5/5 still passes

If the post-parallel build still fails for the same race reason, that's a separate infrastructure concern (the optimize-images script needs better idempotency under concurrent invocations) ŌĆö track in a future hardening pass.
